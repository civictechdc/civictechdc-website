# Civic Tech DC Website Guide

> **Note for maintainers:** This guide is intended to be linked from `README.md`. It is the
> primary reference for anyone who wants to understand how the site works and how to make
> common changes.

---

## Who This Guide Is For

You do not need to be a professional developer to contribute to this site. If you can edit a
text file and run a few commands in a terminal, you can add a project, write a blog post, or
update copy. This guide walks through everything step by step.

---

## The Technology Stack

Several tools work together to produce the finished website. Here is a plain-language tour of
each one.

### Jekyll

[Jekyll](https://jekyllrb.com/) is the core of the site. It reads your content files
(written in Markdown) and your templates, then combines them into a folder of plain HTML
files that a web server can serve directly. There is no database and no server-side code at
runtime. The whole site is pre-built.

Jekyll is written in Ruby, which is why you need Ruby installed to run it locally.

### Liquid

Liquid is the templating language Jekyll uses inside HTML files. You will see it in
`_layouts/` and `_includes/` as tags like `{{ page.title }}` and
`{% include components/project-cards.html %}`. It lets templates pull in data and reuse
components without repeating HTML everywhere.

### USWDS (U.S. Web Design System)

[USWDS](https://designsystem.digital.gov/) is a design system created by the US government.
It provides pre-built, accessible components (buttons, cards, navigation, grids) along with
fonts and icons. This site uses USWDS for its visual foundation so the site is consistent and
accessible out of the box. The USWDS styles live in `assets/css/` after compilation.

### Sass

Sass is a stylesheet language that compiles down to regular CSS. It supports variables,
nesting, and imports, which makes large stylesheets easier to manage. The site's custom Sass
lives in `sass/`. You edit those files and then run `npm run build` to produce the final CSS
in `assets/css/`.

### Gulp

[Gulp](https://gulpjs.com/) is a build-automation tool. The site uses it for two main jobs:

1. **Compile styles.** Gulp runs the USWDS compiler (which processes Sass) and then adds a
   cache-busting hash to each CSS filename.
2. **Process images.** Gulp takes source images from `_images/` and generates multiple
   resized versions in `assets/images/`. This is what powers the responsive images
   (`srcset`) that load quickly on any screen size.

You do not interact with Gulp directly in most cases. `npm run build` runs it for you.

## Repository Map

```
civictechdc-website/
├── _config.yml          # Jekyll configuration: site name, URL, plugins, collections
├── _layouts/            # Page templates (base HTML shell, default, hero, project, post)
├── _includes/           # Reusable HTML components (header, footer, cards, image tag)
│   ├── core/            # Site-wide structural pieces (nav, meta, CSS link tags)
│   └── components/      # Page-level building blocks (project cards, event cards, CTAs)
├── _projects/           # One Markdown file per project (shows up on the Projects page)
├── _events/             # One Markdown file per event (shows up on the Events page)
├── _posts/              # Blog posts, named YYYY-MM-DD-slug.md
├── _images/             # SOURCE images: put originals here, not in assets/
├── sass/
│   ├── custom/
│   │   └── styles.scss  # Main custom stylesheet (edit this to change site styles)
│   └── theme/           # USWDS theme settings (colors, fonts, spacing tokens)
├── assets/              # BUILD ARTIFACTS: do not hand-edit anything in here
│   ├── css/             # Compiled CSS (generated by gulp)
│   ├── js/              # JavaScript (generated by USWDS / gulp)
│   ├── fonts/           # Fonts (generated by USWDS / gulp)
│   ├── images/          # Processed images (generated by gulp from _images/)
│   └── img/             # USWDS icons (generated by gulp)
├── _data/
│   └── css-manifest.json # Maps CSS filenames to hashed versions (generated by gulp)
├── _site/               # The compiled website output (never touch this)
├── index.md             # Homepage content
├── about.md, events.md, projects.md, blog.md, ...  # Top-level page content files
├── gulpfile.js          # Gulp task definitions
├── package.json         # Node.js dependencies and npm scripts
├── Gemfile              # Ruby/Jekyll dependencies
└── .tool-versions       # Tells asdf which Ruby and Node versions to use
```

The most important mental model: **you write content in Markdown and Sass; the build tools
produce everything under `assets/` and `_site/`.**

## Setting Up Your Local Environment

### Prerequisites

You need three things installed before you start:

1. **Git** - for cloning the repo and creating branches.
   [Download Git](https://git-scm.com/downloads).
2. **asdf** - a version manager that installs the correct Ruby and Node versions
   automatically. [Install asdf](https://asdf-vm.com/guide/getting-started.html).
3. A **terminal** (Terminal on macOS, Windows Terminal or WSL on Windows, any terminal on
   Linux).

### Step-by-step setup

**1. Fork and clone the repo.**

On GitHub, click "Fork" in the top-right corner of the
[civictechdc/civictechdc-website](https://github.com/civictechdc/civictechdc-website)
repository. Then clone your fork to your computer:

```bash
git clone git@github.com:YOUR-GITHUB-USERNAME/civictechdc-website.git
cd civictechdc-website
```

**2. Install Ruby and Node via asdf.**

The `.tool-versions` file in the repo root tells asdf which versions to use (currently Ruby
3.2.0 and Node 23.7.0). Run these commands inside the project directory:

```bash
asdf plugin add ruby https://github.com/asdf-vm/asdf-ruby.git
asdf plugin add nodejs https://github.com/asdf-vm/asdf-nodejs.git
asdf install
```

If you are on Ubuntu and the Ruby install fails, try these first:

```bash
sudo apt install libyaml-dev libffi-dev
```

**3. Install Ruby gem dependencies.**

```bash
bundle install
```

**4. Install Node dependencies.**

```bash
npm install
```

**5. Start the development server.**

```bash
npm run serve
```

This command starts Gulp (which watches `_images/` and `sass/` for changes) and Jekyll (which
watches your content files). Open a browser to [http://localhost:4000](http://localhost:4000).
The page reloads automatically when you save changes.

## Making Common Changes

### Edit page copy

Most pages are Markdown files at the root of the repo: `index.md`, `about.md`, `projects.md`,
`events.md`, `community.md`, etc. Open the file you want to change and edit the text below the
`---` frontmatter block. Save the file and the dev server will refresh automatically.

**Example:** to change the intro text on the About page, open `about.md` and edit the
paragraph text.

### Add or update a project

Projects are Markdown files in `_projects/`. Each file has a frontmatter block at the top
with metadata.

**To add a new project**, create a new file named `_projects/your-project-name.md`:

```markdown
---
layout: project
is_active: true
title: Your Project Name
image: project_thumbnails/your-image.png
image_alt_text: Describe what is visible in the image
description: One sentence describing what the project does.
---

Write a longer description of the project here in Markdown. This appears on the project's
detail page.
```

- `is_active: true` means the project appears under "Active Projects" on the homepage and
  projects page.
- `is_active: false` moves it to "Past Projects."
- The `image` path is relative to `_images/`. Put the source file there and run
  `npm run build` to generate the processed version (see the Images section below).
- The title, description, image, and image alt text also supply the project's search and
  social-sharing metadata.

**To mark a project as inactive**, open its `.md` file and change `is_active: true` to
`is_active: false`.

### Add an event

Events are Markdown files in `_events/`. Create a new file named after your event, e.g.
`_events/my-event-name.md`:

```markdown
---
layout: events
title: "My Event Title"
date: 2026-07-15
location: "Washington, DC"
card_image: events/my-event-image.png
social_image_alt: Describe what is visible in the event image
description: "A brief description of the event."
redirect_to: https://lu.ma/your-event-link
sitemap: false
---
```

- `redirect_to` sends the visitor to the Luma or Eventbrite page. Keep `sitemap: false` on
  redirect-only entries so search engines index the destination instead.
- For a hosted event page, omit `redirect_to` and copy the `event` structured-data block
  from an existing hosted event. Its date, time, venue, and address must match the visible
  page.
- The `card_image` path is relative to `_images/`. Add the source in `_images/events/` and
  run `npm run build`.

### Add a blog post

Blog posts live in `_posts/`. The filename must follow the pattern
`YYYY-MM-DD-your-post-slug.md`:

```markdown
---
layout: post
title: "Your Post Title"
date: 2026-07-15 10:00:00 -0500
author: "Your Name"
categories: [Community, Announcements]
tags: [civic tech, community]
excerpt: "A one-sentence teaser that appears on the blog listing page."
social_image: representative-image.jpg
social_image_alt: Describe what is visible in the image
---

Your post content goes here in Markdown.
```

- `categories` and `tags` power the archive pages at `/blog/category/...` and
  `/blog/tag/...`.
- `excerpt` is shown on the blog index. If you leave it out, Jekyll uses the first paragraph.

### Add or update an image

The site uses a responsive-image system. **Never place images directly in `assets/images/`
by hand.** Instead:

1. Put your original image file in `_images/` (in the appropriate subfolder, e.g.
   `_images/project_thumbnails/` or `_images/events/`).
2. Run `npm run build`. Gulp will generate multiple resized versions in `assets/images/`
   automatically (400w, 800w, 1200w, 1800w for JPG/PNG files; SVG, GIF, and WebP are passed
   through unchanged).
3. Reference the image in your Markdown or HTML using the `responsive-image` include:

```liquid
{% include responsive-image.html src="project_thumbnails/my-image.png" alt="Describe the image here" %}
```

The `src` value is relative to `_images/`. The include handles all the `srcset` markup for
you.

**Do not write `<img>` tags that point directly into `assets/images/`.** Use the include
instead so the responsive variants and lazy-loading are set up correctly.

### Change styles

Custom CSS lives in `sass/custom/styles.scss`. Edit that file, then rebuild:

```bash
npm run build
```

This runs Gulp, which compiles the Sass files into `assets/css/styles.css` (with a
cache-busting hash). Commit both your changes in `sass/custom/styles.scss` and the
regenerated files under `assets/css/` and `_data/css-manifest.json`.

For design tokens (colors, spacing, font sizes), look for the `:root { ... }` block near the
top of `sass/custom/styles.scss`. USWDS theme settings (which affect the design-system
components directly) are in `sass/theme/_uswds-theme.scss`.

## The Build-Artifact Rule

Everything under `assets/` and `_site/` is **generated output**. These files are the result
of running Gulp on your source files. They are committed to the repo so that GitHub Pages can
serve them without needing a build step, but they should never be hand-edited.

**Why?** If you edit a file in `assets/css/` directly, your change will be invisible the next
time anyone runs `npm run build`, because Gulp overwrites those files from the sources in
`sass/`. Worse, the CI pipeline runs `gulp compile` on every pull request and checks whether
it produces any uncommitted changes. If your hand-edit differs from what Gulp would generate,
the CI check fails and the PR is blocked.

**The rule in practice:**

| What you want to change | Edit this                                                 | Then run                     |
| ----------------------- | --------------------------------------------------------- | ---------------------------- |
| Site styles             | `sass/custom/styles.scss`                                 | `npm run build`              |
| USWDS theme tokens      | `sass/theme/_uswds-theme.scss`                            | `npm run build`              |
| Images                  | `_images/`                                                | `npm run build`              |
| Page content            | `*.md` files at root, `_projects/`, `_events/`, `_posts/` | nothing (Jekyll picks it up) |
| Templates               | `_layouts/*.html`, `_includes/**/*.html`                  | nothing (Jekyll picks it up) |

After running `npm run build`, commit all changed files including the regenerated artifacts.

## Linting and Formatting

The project uses [Prettier](https://prettier.io/) for consistent formatting. Format the files
you changed, then inspect the diff:

```bash
npx prettier --write path/to/file.md
```

`npm run lint` formats the entire repository. Use it cautiously: the Liquid plugin can reflow
complex include arguments incorrectly. CI runs Prettier in check mode and fails on formatting
drift; complex Liquid partials that cannot be formatted safely are listed in `.prettierignore`.

## SEO and Metadata Checks

Before opening a pull request that changes content, routes, images, or templates, run:

```bash
npm run check:seo
```

This builds the production site in a temporary directory and checks every rendered page,
redirect, social preview, structured-data block, sitemap entry, robots directive, and feed
entry. It also catches duplicate or overlong search metadata.

## Contributing

Here is the high-level flow:

1. **Open an issue** (or find an existing one) to describe what you want to change.
2. **Fork the repo** and create a branch off `main` with a descriptive name, e.g.
   `add-new-project-my-tool`.
3. **Make your changes**, following the steps above.
4. **Run Prettier on the files you changed** and inspect the formatting diff.
5. **Run `npm run build`** if you changed anything in `sass/` or `_images/`.
6. **Run `npm run check:seo`** if you changed content, routes, images, or templates.
7. **Commit your changes** (including any regenerated assets).
8. **Open a pull request** against `civictechdc/civictechdc-website:main` with a clear title
   and description.
9. **CI checks run automatically.** If anything fails, fix it and push
   again.
10. A maintainer reviews and merges. The site deploys automatically to
    [civictechdc.org](https://www.civictechdc.org) after merge.

For the full step-by-step contribution instructions, see [CONTRIBUTING.md](https://github.com/civictechdc/civictechdc-website/blob/main/CONTRIBUTING.md).

If you get stuck, ask in the **#civictechdc-website** channel on the
[Civic Tech DC Slack](https://www.civictechdc.org/slack.html). We're happy to help.
